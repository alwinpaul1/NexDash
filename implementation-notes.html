P2.0 · What "Implement Dispatcher Console / Route Planning" means here open

I read this as: implement the agreed all-factors route optimiser — order stops using real road distances, real per-leg terrain & wind, the trained ML energy model, and payload decay; then run the winning order through the charge-/EU-561-/deadline-aware planner with two new constraints: the truck must be able to reach a charger from the final destination, and optionally return to origin. Wired into the existing Routes console. Open: if "Dispatcher Console" implies a broader UI rebuild (not upgrading the existing Optimize flow), confirm and I'll widen scope.

P2.1 · Objective

Single-vehicle visiting order minimising cost = energy_kWh × €/kWh + driving_time_h × €/h, on real road distances/times, with per-leg energy from the ML model using real gradient + wind + set-determined payload; then a charge-aware feasibility plan with the two new constraints.

P2.2 · Architecture provisional — pending code map

Driving constraint: the TomTom key is a browser key; the backend holds the ML model + server-side Open-Meteo. So:

TierResponsibility
Frontend (routePlanner.js + planner UI)Build the real N×N road distance/time matrix (TomTom); find the nearest charger to the final destination; collect inputs (round-trip toggle, per-stop weights/deadlines, temp); POST to /api/optimize; loading state; render order + constraint verdicts.
Backend (optimizer.py, route_planner.py, /api/optimize)Receive matrix + node coords. Enrich each node with elevation/wind (geodata). Score each leg's energy via predict_energy (real dist + endpoint-derived gradient + wind + payload-for-set). Held-Karp / 2-opt over real costs. Plan charging + EU-561 + constraints on the chosen order. Return order + plan + verdicts + assumptions.

P2.3 · Design decisions

IDDecisionRationale
P2-D1 provPer-leg gradient from endpoint elevation delta ((elev_j−elev_i)/dist_ij), not full per-leg polyline geodata.Full per-leg geodata is O(N²) calls; per-node elevation is O(N) and gives a defensible ordering grade. The final displayed plan still uses full real geometry/elevation.
P2-D2ML energy memoised per (leg, payload-bucket).Held-Karp evaluates leg cost millions of times for ~11 stops; only payload varies per leg and it's set-determined → bucketise to bound sklearn calls.
P2-D3 provMatrix via TomTom Matrix Routing if entitled, else N² calculateRoute.EV-Search was 403 on this key, so Matrix entitlement is unknown; calculateRoute is proven (Tier-A).
P2-D4Charging is a feasibility layer on the chosen order, not jointly co-optimised.True E-VRPTW is NP-hard; this is a strong, honest approximation, surfaced in assumptions[].
P2-D5 prov"Reach charger from final stop": require arrival SOC ≥ energy(final → nearest charger) + min_soc; red verdict if violated.Matches project ethos (honest disclosure, conservative verdict). Nearest charger from the TomTom POIs the frontend already fetches.
P2-D6Round-trip reuses the existing return_to_origin path; new work is the UI toggle + charging across the return leg.Smallest honest change; avoids re-implementing supported behaviour.

P2.4 · Deviations

  • Ordering now uses real road distance/time (replaces the great-circle×1.3 proxy for the order choice, not just the final plan).
  • "All factors" is approximated: terrain enters ordering as an endpoint-derived representative gradient (P2-D1); charging is decoupled from ordering (P2-D4).
  • The earlier "freeze" decision was intentionally reversed at user direction.

P2.5 · Tradeoffs

  • Exact E-VRPTW vs decoupled order→charge: chose decoupled (NP-hard otherwise; not warranted for a bonus).
  • Full per-leg geodata vs endpoint grade: chose endpoint grade to keep API calls O(N) — coarser ordering terrain for tractable latency/quota.
  • Backend-calls-TomTom vs frontend-builds-matrix: frontend builds it (the key is a browser key); optimisation stays on the backend with the model.
  • ML vs physics for ordering cost: chose ML per request (memoised); ML ≈ physics by construction, so the value is consistency with the graded model, not better orders.
